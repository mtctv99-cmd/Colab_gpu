# Kế hoạch tối ưu hóa và nâng cấp hệ thống Server TTS & LLM

Tài liệu này chi tiết hóa kế hoạch sửa lỗi và tối ưu hóa hệ thống Backend (FastAPI) và Worker (Google Colab).

---

## 1. Các tệp tin ảnh hưởng chính (Critical Files)
- `app/database.py` (Cơ chế migration)
- `app/services/auth.py` (Xử lý balance / race condition)
- `app/routes/auth.py` (Xác thực API Key / Whitelist IP / Expiration)
- `app/routes/openai.py` (Xử lý stream LLM / Hoàn tiền / Hủy task)
- `app/routes/ws.py` (WebSocket heartbeat / validation)
- `app/orchestrator/lifecycle.py` (Database session / scale checking)
- `colab/worker.py` (GPU VRAM cache eviction / reconnect loop exit 4002)
- `colab/llm_worker.py` (Async thread execution / cancel task / ping handle)
- `frontend/src/app/dashboard/page.tsx` (Dashboard UI & LLM Playground tab)

---

## 2. Chi tiết kế hoạch tối ưu & nâng cấp

### PHẦN A: BẢO MẬT & ĐỒNG BỘ SỐ DƯ (Security & Concurrency)

#### 1. Bảo mật API Key (`app/routes/auth.py` & `app/routes/openai.py` & `app/routes/tts.py`)
- **Vấn đề:** API Key có các cấu hình `allowed_ips`, `expires_at`, `rate_limit` nhưng hàm `get_current_user` hiện tại chỉ check hash và status hoạt động.
- **Giải pháp:**
  - Sửa đổi hàm xác thực API Key để kiểm tra hạn sử dụng (`expires_at`).
  - Phân tích IP client từ request (đọc `request.client.host` hoặc header `X-Forwarded-For`).
  - Parse `allowed_ips` từ JSON trong database và kiểm tra xem IP client có khớp không (nếu có whitelist).
  - Tích hợp thêm rate limiting dựa trên cấu hình của khóa.

#### 2. Ngăn ngừa Race Condition số dư (`app/services/auth.py`)
- **Vấn đề:** Thay đổi số dư bằng gán trực tiếp (`user.balance -= cost`) dẫn đến tranh chấp ghi đè số dư (race condition) khi gọi song song nhiều API.
- **Giải pháp:**
  - Sử dụng SQL Update trực tiếp:
    ```python
    await db.execute(
        update(User)
        .where(User.id == user_id)
        .values(balance=User.balance - amount)
    )
    ```
  - Hoặc sử dụng cơ chế khóa dòng `with_for_update()` khi lấy thông tin user để cập nhật số dư.

---

### PHẦN B: ỔN ĐỊNH LIFECYCLE & HẠ TẦNG (Lifecycle & Orchestrator)

#### 3. Tránh nghẽn DB Connection Pool trong Orchestrator (`app/orchestrator/lifecycle.py`)
- **Vấn đề:** Hàm `scale_check` gọi `worker_manager.launch()` chiếm dụng DB session trong thời gian chạy mạng đồng bộ (2-3 phút), làm cạn kiệt connection pool.
- **Giải pháp:**
  - Thay đổi scope của DB session. Chỉ truy vấn DB để lấy account và thông tin cấu hình, giải phóng session (commit/close hoặc thoát khối `async with`).
  - Gọi hàm `launch()` độc lập bên ngoài DB block. Sau khi launch thành công/thất bại, mở một DB session mới ngắn hạn để cập nhật trạng thái runtime.

#### 4. Khắc phục lỗi Colab reconnect bị từ chối (`app/routes/ws.py`)
- **Vấn đề:** Khi server restart, worker cũ bị set sang `WS_LOST`. Khi worker kết nối lại, server từ chối vì chỉ chấp nhận session `WS_ALIVE` hoặc `WS_STARTING`.
- **Giải pháp:**
  - Sửa `validate_worker_registration()` để chấp nhận trường hợp reconnect của session có trạng thái `WS_LOST` nếu định danh `worker_session_id` khớp, phục hồi trạng thái thành `WS_ALIVE` thay vì đóng websocket với mã `4002`.

#### 5. Sửa lỗi check Heartbeat & Requeue Task (`app/routes/ws.py`)
- **Vấn đề:** Kiểm tra sai trạng thái tài khoản (`acc.status == "ACTIVE"`) khiến server không phát hiện worker chết và không đẩy task dở dang lại vào queue.
- **Giải pháp:**
  - Sửa hàm check heartbeat: Chỉ cần kiểm tra heartbeat ping cuối. Nếu quá hạn, đánh dấu tài khoản là `OFFLINE` và thực hiện requeue tất cả task đang gán cho worker đó (bất kể `status` của account là gì).

### PHẦN C: KHẮC PHỤC LỖI TẠI COLAB WORKER (Worker Stability)

#### 6. Cơ chế tự hủy của Worker khi bị Server reject (`colab/worker.py` & `colab/llm_worker.py`)
- **Vấn đề:** Worker bị server đóng websocket bằng mã `4002` (Unauthorized) vẫn tiếp tục cố gắng reconnect vô hạn.
- **Giải pháp:**
  - Trong vòng lặp kết nối WS của worker, nếu bắt được exception đóng socket với code `4002`, thực hiện in lỗi chi tiết và gọi `sys.exit(1)` lập tức. Điều này giúp tiến trình quản lý phía Colab biết worker bị từ chối vĩnh viễn và khởi động lại phiên làm việc mới sạch sẽ.

#### 7. Eviction Cache tránh CUDA Out of Memory (`colab/worker.py`)
- **Vấn đề:** `_voice_prompt_cache` lưu trữ không giới hạn các tensor GPU của file âm thanh mẫu.
- **Giải pháp:**
  - Triển khai cơ chế cache LRU (Least Recently Used) đơn giản. Giới hạn cache tối đa 10-20 voice prompts. Khi vượt quá, thực hiện xóa tensor cũ khỏi bộ nhớ GPU (`del tensor`) và gọi `torch.cuda.empty_cache()`.

#### 8. Sửa nghẽn loop WebSocket ở LLM Worker (`colab/llm_worker.py`)
- **Vấn đề:** Luồng xử lý LLM chạy đồng bộ chặn vòng lặp nhận tin nhắn WS chính, làm mất ping và không thể nhận lệnh Cancel.
- **Giải pháp:**
  - Đẩy tiến trình sinh text sang một thread/task riêng biệt.
  - Vòng lặp WS chỉ nhận tin nhắn, phân loại lệnh và giao tiếp với task xử lý qua `asyncio.Queue` hoặc biến trạng thái để có thể phản hồi ping lập tức.
  - Sửa lỗi LLM Cancel: Nếu nhận được tín hiệu hủy, dừng tiến trình generator ngay lập tức và gửi gói tin hủy chính xác thay vì gửi gói tin `COMPLETED`.

---

### PHẦN D: NÂNG CẤP GIAO DIỆN KIỂM THỬ LLM CHAT PLAYGROUND (Frontend Update)

#### 9. Giao diện Chat Playground cho người dùng thử nghiệm model LLM
- **Vấn đề:** Giao diện dashboard hiện tại chỉ có tab thử nghiệm TTS, chưa có tab để gọi API `/v1/chat/completions` để chat và kiểm tra model LLM (như chọn parameters, stream completion).
- **Giải pháp:**
  - Bổ sung sidebar item "Chat Playground" trong `frontend/src/app/dashboard/page.tsx`.
  - Triển khai giao diện Chat hoàn chỉnh:
    - Bảng điều chỉnh System Prompt (bên trái).
    - Cửa sổ tin nhắn dạng Chatbox (ở giữa) hỗ trợ streaming text (SSE) phản hồi trực tiếp khi model trả về từng chunk và có nút Cancel để huỷ sinh văn bản giữa chừng.
    - Cột tuỳ biến tham số: chọn model, chỉnh `temperature` (thông qua slider), chỉnh `max_tokens` (thông qua slider).

## 3. Kế hoạch xác thực kiểm thử (Verification Plan)

### Kiểm thử tự động (Unit / Integration Tests)
1. **Kiểm tra API Key Auth:** Gửi request kèm API key hợp lệ, hết hạn, và IP ngoài whitelist. Xác nhận mã HTTP trả về đúng (200, 401, 403).
2. **Kiểm tra Race Condition số dư:** Sử dụng công cụ benchmark (như `locust` hoặc script Python chạy song song `httpx.AsyncClient`) gửi 20 request đồng thời để trừ tiền. Xác nhận số dư cuối cùng giảm chính xác bằng tổng số tiền của 20 request, không bị lệch.

### Kiểm thử luồng thực tế (End-to-End Tests)
1. **Mô phỏng Mất mạng Worker:** Chạy worker, ngắt kết nối mạng của worker trong 45 giây. Kiểm tra xem Server có chuyển worker sang `OFFLINE` và requeue task hay không. Khi mạng có lại, kiểm tra worker kết nối lại thành công (không bị lỗi `4002`).
2. **Mô phỏng Hủy task LLM:** Gửi request stream chat completion dài, ngắt kết nối client nửa chừng. Xác nhận Server gửi lệnh `cancel_task` xuống worker, worker dừng chạy LLM ngay lập tức, hoàn trả đúng số dư còn lại cho user.
3. **Mô phỏng Chat Playground UI:** Đăng nhập vào dashboard, mở tab Chat Playground, nhập nội dung tin nhắn và gửi. Xác nhận tin nhắn model stream phản hồi đúng từng chữ một trên UI, kiểm tra thanh trượt `temperature` và `max_tokens` truyền đúng param lên Server.
